# Implementation Plan: Moderation Panel

**Branch**: `main` | **Date**: 2026-06-24 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `/specs/007-moderation-panel/spec.md`

## Summary

Build the Moderation Panel module for the On The Way Admin Dashboard. This replaces the current `/moderation` stub with a full three-section page: **Flagged Reports** (road reports marked as false, spam, or duplicate — with Approve / Remove / Warn User actions), **Suspicious Users** (accounts flagged for unreliable behavior — with Warn / Suspend / Flag to Admin actions), and a **Pending Moderation Actions** queue with priority-tagged items and "Review" navigation. A header badge shows the live total pending count. All sections auto-refresh via background polling (~45 s). Remove (report) and Suspend (user) require a single confirmation step before executing. Flag to Admin triggers an active notification to super-administrators server-side.

## Technical Context

**Language/Version**: TypeScript ~6.0 (strict mode)  
**Primary Dependencies**: React 19, Vite, TanStack React Query v5, React Router v7, Axios, Shadcn UI, Radix UI, Tailwind CSS v4, Zod, Lucide React, Sonner  
**Storage**: N/A (admin read + action mutations; no browser persistence beyond URL state)  
**Target Platform**: Web browser (admin dashboard, desktop-first, 1024px+)  
**Project Type**: Web application (admin dashboard)  
**Performance Goals**: Panel sections visible within 3 s of navigation; action feedback within 3 s of click; pending count updates within 1 s of action success; skeleton content within 200 ms of page access  
**Constraints**: No new runtime dependencies; no `useEffect` for data fetching; no confirmation modal for reversible actions (Approve, Warn User, Warn, Flag to Admin); single AlertDialog confirmation for Remove and Suspend; background polling with `refetchInterval: 45_000`; independent section failure/recovery  
**Scale/Scope**: Bounded lists (no pagination in Phase 7); polling-based freshness; fixture fallback pattern for backend integration

## Constitution Check

All components are functional, scoped to moderation workflows, and kept to single-responsibility cards/sections. No `any` types are required.
- **Data and State**: PASS. Server data flows through four independent React Query hooks. The shared Axios client is used for all API calls. Mutations use `useMutation` with query invalidation on success. No `useEffect` for fetching. Sonner toasts provide success/error feedback. No global state is needed.
- **UX Consistency**: PASS. UI uses Shadcn AlertDialog (for confirmations), Badge (for flag reasons, warned state, priority), Skeleton (loading states), and card-based layouts. All action buttons are keyboard-accessible. Empty and error states are explicit per section.
- **Performance**: PASS. Background polling uses `refetchInterval: 45_000` with `refetchIntervalInBackground: false`. Sections load and fail independently so a single bad response does not degrade the full panel. No new dependencies or bundle growth are introduced. Lists are bounded in Phase 7; no pagination/virtualization complexity required.
