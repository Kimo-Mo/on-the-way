# Implementation Plan: Users Management

**Branch**: `004-users-management` | **Date**: 2026-06-18 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/004-users-management/spec.md`

## Summary

Build a fully functional Users Management module for the On The Way Admin Dashboard. This replaces the current stub at `/users` with a server-paginated, searchable, filterable users table and adds a new `/users/:id` details page showing a user's complete profile and activity history. Table state (pagination, search, filters) is persisted via URL search parameters. All data is fetched via React Query hooks backed by the shared Axios client, with typed fixture fallbacks until the live .NET backend is connected.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode)
**Primary Dependencies**: React 18, Vite, TanStack React Query v5, React Router DOM v7, Axios, Shadcn UI, Radix UI, Tailwind CSS, Zod, React Hook Form, Lucide React
**Storage**: N/A (read-only views for Phase 4; no write operations)
**Target Platform**: Web browser (admin dashboard, desktop-first)
**Project Type**: Web application (admin dashboard)
**Performance Goals**: Users table renders and becomes interactive within 2 seconds of navigation; filter/search UI responds within 300ms of user input
**Constraints**: Server-side pagination required (do not fetch all users at once); no avoidable duplicate API requests; no new runtime dependencies introduced
**Scale/Scope**: Supports up to thousands of users; all filtering and pagination delegated to the backend

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **Code Quality** ✅: All types are explicit (`User`, `UserDetails`, `UsersQueryParams`, `PaginatedResponse<T>`). Components are small and functional. No `any` types permitted. Data model is defined in `data-model.md`.
- **Data and State** ✅: Server data fetched exclusively via `useUsers` and `useUserDetails` React Query hooks, backed by the shared Axios client. URL search params (via `useSearchParams`) are the sole source of truth for table filter/pagination state — no Zustand for this feature. No `useEffect` for fetching.
- **UX Consistency** ✅: All components use Shadcn/Radix/Tailwind patterns. Skeleton loading rows, inline empty state with "Clear Filters" action, and destructive Alert for errors are explicitly planned. Keyboard focus, visible focus states, and accessible table markup are required.
- **Performance** ✅: Server-side pagination enforced via query params. `placeholderData: keepPreviousData` prevents layout flashes. Stale times set to minimize redundant refetches. Route-level code splitting via React Router lazy loading can be applied to `UserDetails` if bundle analysis warrants it.

*Post-design re-check*: All design decisions in `research.md` and `data-model.md` pass the above gates. No violations requiring justification.
